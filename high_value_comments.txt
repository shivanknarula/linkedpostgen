One thing I’ve learned building my own ROS2 robot is that nothing ever goes 100% to plan, why? I didn’t want to buy a pre-built turtle bot robot and figured this would let me dip in all areas of engineering and guess what. It did. 
When I first set out to create a ROS 2 Humble-based mobile robot, the vision was clear: 
Raspberry Pi 4 for brains, Arduino with encoders for odometry, and an LD14 LiDAR to map the world (I quickly found out LIDAR sensors aren’t so cheap). On paper? Simple. In practice? A real engineering adventure.
At the start, I had to “fake” odometry just to get SLAM Toolbox running. Watching the LiDAR draw a live map reminded me of building a ROS1 robot during my university days. But the real breakthrough came when I finally got encoder-based odometry working—suddenly the robot wasn’t guessing, it knew where it was.
Then came the plot twist: the base literally cracked under testing, I decided to get stronger motors which were x4 heavier and the shaft was much larger. At first it felt like a disaster, but it turned into one of the best parts of the project. I fired up the 3D printer, redesigned the mountings, and ended up with a stronger, modular platform that made iterating so much easier. (As well as some help off the time help from friends to make the new shaft sleeves) 
Since then, I’ve integrated:
	•	SLAM Toolbox for real-time mapping
	•	Nav2 stack for autonomous path planning and obstacle avoidance
	•	Explore_Lite for full autonomous exploration
	•	A web dashboard for remote monitoring and control
It’s been a crash course in ROS2 and seeing all the changes —balancing low-level hardware with high-level autonomy. Every problem, from TF transform errors to busted parts, became an opportunity to learn, rebuild, and move forward.
📚 Along the way, I’ve been reading “A Concise Introduction to Robot Programming with ROS2” by Francisco Martín Rico (Though I accidentally bought the first version). The book has been an incredible companion to the build, helping me connect the theory with the hands-on work I was doing. I’m still reading and learning from it now, and it continues to push my understanding forward.
Also the change in the URDF using comma separated links in the definition was a really annoying change but I can see the benefit in hindsight. Open Robotics  
Excited to keep pushing this platform further — and to connect with others working in ROS 2, autonomous navigation, or robotics integration.

This week I spent some time working through a robotics learning repository focused on perception and navigation in ROS2.
I followed a series of exercises from the open source project robovison_ros2 by (ARTenshi). The repository walks through different steps of working with RGB-D sensors, point clouds, and robot navigation using ROS2.
While working through the exercises I explored things like:
- Reading RGB-D data and point clouds from ROS bag 
- Extracting useful spatial information from the cloud like central points and distances
- understanding how perceptions data can be used for robot decisions 
- Building a simple reactive navigation behaivor
One of the interesting parts of the exercises was connecting perception with motion, using sensor data to determinate whether the robot should move forward or rotate to avoid obstacles.
Thanks to the author for sharing the repository and exercises openly.
